Reviewer evidence strategy

Hard evidence beats persuasion.

I am a one man development team and agentic development expert. I do more, with less by turning agent throughput into public artifacts, replayable checks, and risk-bearing claims that reviewers can inspect.

Research basis

What makes a signal worth attention.

In job-market signaling, the receiver cannot directly observe the candidate's real productivity. Useful signals reduce that uncertainty because they are visible and expensive to imitate. The strongest portfolio signal is therefore a public work sample with a clear verification path.

Evidence filterPortfolio translation
Observable A reviewer can see the artifact, command, commit, input, output, or limitation without private context.
Differentially costly The evidence is much harder for a weak builder to imitate: tests, proofs, failure cases, replay scripts, and claim discipline.
Decision-relevant The signal maps to a hiring or collaboration decision: can this person build, verify, explain, and ship?
Risk-bearing The claim can fail in public. A broken replay, false metric, or weak proof surface should be discoverable.

This page converts signaling theory into an operating system for the portfolio: fewer claims by assertion, more claims bound to inspectable evidence.

Evidence stack

Reviewer-grade evidence I should keep adding.

These are high-signal mechanisms because each one makes evaluation cheaper for the receiver and more expensive for an imitator.

01 / VERIFY files 1 cmd

Each public repo should expose a short VERIFY.md: clone, install, run focused tests, run proof checks, regenerate metrics, and compare expected output.

02 / replay receipts hash

A receipt binds commit, command, environment, input, output summary, runtime, and SHA-256 artifact hashes. This turns a claim into something rerunnable.

03 / break-me queue public

GitHub issues labeled break-me, verify-me, and replicate invite reviewers to attack specific claims instead of reading a whole repo cold.

04 / failure ledger limits

Known weaknesses, refuted claims, failed proof attempts, fixed counterexamples, and open assumptions should be visible. Negative evidence raises trust.

05 / work sample menu 4 hr

A hiring team should be able to assign a focused review challenge: find a boundary gap, add a replay test, tighten a claim, or explain a proof surface.

06 / demo receipts clip

Short terminal recordings should be linked to the exact command and commit they demonstrate. The clip is marketing; the command is the signal.

07 / third-party reruns names

Independent rerun notes from reviewers, collaborators, or employers become higher-value evidence than self-reported claims.

08 / claim badges scoped

Badges should describe evidence status: replayable, third-party-rerun, counterexample-fixed, proof-surface, bounded-claim.

Method terminology

How I manage agentic development.

My signal includes the management layer that turns untrusted agent labor into validated artifacts. These terms define the method I want reviewers to recognize.

Verifier-bound agentics

Agents generate candidates; deterministic checks, proof attempts, tests, and human review decide what survives.

Claim-carrying work packets

Each agent task carries a claim, boundary, allowed evidence, failure mode, and promotion rule before implementation starts.

Adversarial context management

Reviewer agents and scripts attack assumptions, search for counterexamples, and force limitations into the public artifact.

Receipt-first promotion

A change earns status through a durable receipt: command, result, metric, theorem path, regression test, or documented counterexample.

Evidence upgrades

Replace weak claims with stronger evidence.

The current site already has quantitative metrics, case studies, and public links. The next quality jump is to make the review path feel like a lab notebook: each claim has a receipt and each limitation is findable.

Lower signalHigher signal
"I built a large codebase." Generated LOC data, exclusion rules, source script, commit hash, and rerun command.
"I use formal methods." Theorem file paths, proof commands, expected failures, assumptions, and machine-check status.
"I am good with AI." Agent packet templates, reviewer transcripts, deterministic acceptance gates, and rejected outputs.
"This project is valuable." Comparable market context, conventional build-cost estimates, reusable artifacts, and explicit valuation caveats.
"This is reliable." Replay receipts, tamper tests, negative cases, failure ledger, and public issue challenges.

Reviewer paths

Make evaluation fast at three depths.

Different receivers have different time budgets. The site should expose a path for each, with increasingly costly evidence as the review gets deeper.

  1. 10-minute screen

    Read the quantitative value case, inspect one project thumbnail, and compare claims against the evidence matrix.

  2. 60-minute technical review

    Clone one repo, run a focused replay command, inspect the claim table, and read one limitation section.

  3. 4-hour work sample

    Ask me to tighten a claim, add a failing test, explain a proof boundary, or turn an agent-generated change into a receipt-bearing patch.

Immediate execution

Highest ROI moves from here.

If I want more signal quickly, these are the next public artifacts to create. They are deliberately practical because the goal is to reduce reviewer uncertainty.

PriorityArtifactWhy it signals
1 VERIFY.md in MPRD, ZenoDEX, and Formal Methods Philosophy. Turns every major claim into an executable review path.
2 evidence/receipts/*.json for one flagship replay per repo. Creates durable, hash-bound proof that a reviewer can rerun.
3 GitHub issue templates for replication and break-me reviews. Makes public criticism part of the signal instead of a risk to hide.
